# CLAUDE - 東京路線図アプリ プロジェクト概要

## プロジェクト概要
このプロジェクトは東京の電車路線図を表示し、駅間の経路検索機能を提供するWebアプリケーションです。

### 📋 サービス概要
本サイトは利用者が出発駅と到着駅に関連する路線のみを絞って表示できる「**インタラクティブ路線図サービス**」です。推薦システムでは補えない運行イレギュラー時（遅延・運休など）にも、自分で判断できる簡略化された路線図を提供することを目的としています。

### 技術スタック
- **フロントエンド**: Astro + React + TypeScript
- **地図表示**: Leaflet + React-Leaflet
- **開発サーバー**: `npm run dev` (localhost:8080)
- **ビルド**: `npm run build`

### 主要機能
1. **インタラクティブ路線図表示**
   - 39路線の東京電車路線（JR、地下鉄、私鉄、新交通システム）
   - ズームレベル対応駅表示
   - 路線の表示/非表示切り替え
   - 駅クリック時の通過路線情報表示
   - お台場エリア（りんかい線・ゆりかもめ）および首都圏広域（つくばエクスプレス）をカバー
   - 神奈川県方面（南武線・根岸線）の重要路線を追加

2. **経路検索機能**
   - 出発駅・到着駅選択
   - 最適経路推薦（最短時間、乗換回数考慮）
   - 複数経路候補表示

3. **UI機能**
   - 乗換駅のみ表示フィルター
   - 折りたたみ可能な操作パネル
   - レスポンシブデザイン（限定的）
   - **路線表示切替機能**: デフォルト非表示、プログラムで制御可能
     - 状態変数: `showRouteToggleSection` (boolean)
     - 個別路線の表示/非表示切り替え
     - 経路推薦数の設定（1-10件）
     - 全路線表示/非表示の一括切り替え
     - 時間フィルター設定

4. **所要時間フィルター機能（現在実装中）**
   - 出発駅から指定時間内にアクセス可能な駅のみを表示
   - 5分〜60分の範囲で設定可能
   - 乗換を含む最短経路時間で計算
   - デバッグ用の一致性チェック機能付き

5. **駅クリック時刻表ツールチップ**
   - 地図上の駅アイコンをクリックするとツールチップを表示
   - **左カラム**: その駅を通る全路線一覧（経路上の路線は太字＋路線色のボーダーで強調、「乗」ラベル付き）
   - **右カラム**: 選択路線の次発5本（出発時刻・種別・行先）
   - **ルート方向の厳守**: 表示する時刻は選択中の経路の「進行方向」に合致したものに限る
     - `stationTimelineMap` に各駅の経路セグメント情報（`routeKey`, `directionIndex`, `depTime`）を保持
     - 経路上の路線が時刻データなしの場合（例: 小田急江ノ島線）は「乗車路線ですが時刻データなし」と表示し、無関係な路線にフォールバックしない
     - ユーザーが左カラムで非経路路線を選択した場合は「ルート外参考」ラベルを付与
   - **出発時刻設定**: ツールチップ内および駅選択パネルで変更可能（`timetableBaseTime` を共有）
   - ヘッダーに「出発に設定」「到着に設定」ボタンを配置

### 時刻表データ実装方針

**重要**: 時刻表ツールチップの右カラムには「選択中の経路で実際に乗車する路線の、進行方向に合った発車情報」のみを表示する。

- 各路線の時刻データは `src/data/timetableData.ts` の `timetableLines` 配列に登録する
- 各路線の `directions` は `[下り/片方向, 上り/もう一方向]` の2要素配列にすること
- `directionIndex` は `getDirectionIndex(routeKey, fromStation, toStation)` で自動判定される
- 時刻データがない路線（例: 小田急江ノ島線）はツールチップ左列に表示されるが右列はデータなしと表示
- 新路線の時刻データを追加する際は `timetableLines` に push するだけで自動的にツールチップに反映される

**現在時刻データ対応路線** (src/data/timetableData.ts の `timetableLines`):
- 山手線 (`yamanote`)
- 東海道線 (`jrTokaidoMainLine`)
- 中央・総武線 (`chuo`)
- 東京メトロ銀座線 (`ginzaLine`)
- 東京メトロ丸ノ内線 (`marunouchiLine`)
- 東急東横線 (`tokyuToyokoLine`)

## コミットに関する重要な規則

### 1. 確認なしでのコミットを禁止
- **勝手にコミットをしないこと**
- コミットする前には必ずユーザーに確認を求めること
- ユーザーが明示的にコミットを依頼した場合のみ実行すること

### 2. 論理的な単位でのコミット
- **まとめてコミットしない**
- 一つの論理的な変更単位ごとにコミットすること
- 関連のない変更は別々のコミットに分けること
- 各コミットは単独で理解可能で、意味のある変更を含むこと

### 3. コミットメッセージ
- 変更内容を簡潔に説明すること
- 何を変更したかではなく、なぜ変更したかを重視すること

### 4. ブランチとチェックアウトの制限
- **勝手にcheckoutしないこと**
- **勝手にブランチを作成・変更しないこと**
- 現在のブランチで作業を継続すること
- ブランチ操作はユーザーの明示的な指示があった場合のみ実行すること

## 🚫 データに関する絶対ルール

### ヒートマップ・駅統計データは実データのみ使用すること

**AIが生成・推定した値をデータとして使用することを禁止する。**

- `src/data/station-stats-data.json` に保存するデータは、実測・公開データのみ
- 現在の実データソース: **Overpass API（OpenStreetMap）** で収集した以下の6項目のみ
  - `convenienceStoreCount`（コンビニ数）
  - `restaurantCount`（飲食店数）
  - `cafeCount`（カフェ数）
  - `supermarketCount`（スーパー数）
  - `hospitalCount`（病院・クリニック数）
  - `parkAreaM2`（公園面積）
- データがない駅は **灰色（`HEATMAP_NO_DATA_COLOR = '#aaaaaa'`）で表示** する。ゼロや推定値で埋めない
- 新しいフィールドを追加する場合は、**実際のデータソース（API・公開統計）を明記** すること
- 「それっぽい値」「概算」「AIによる推計」は一切使用禁止

**背景:** 2026年6月以前のデータには家賃・乗降客数・治安スコアなどAI生成の推定値が混入していたが、ユーザーが誤った情報に基づいて意思決定することを防ぐため、全て削除した。

## 現在の重要な問題点と解決策

### 🚨 緊急問題: 小田急線の経路推薦が表示されない

**問題の詳細:**
- 藤沢→新宿の経路検索で、小田急線経由のルートが推薦されない
- 理論的には「藤沢（小田急江ノ島線）→ 相模大野 → 新宿（小田急小田原線）」のルートが最適（乗り換え1回）
- 現在はJR東海道線経由のルートのみが推薦されている（乗り換え2回以上）

**現在の調査状況:**
1. ✅ **データファイル確認**: `odakyu-enoshima-line.ts` と `odakyu-line.ts` に藤沢駅・相模大野駅は正しく登録済み
2. ✅ **routes.ts確認**: 小田急線は `routes` オブジェクトに正しく登録済み (`odakyuLine`, `odakyuEnoshimaLine`)
3. ✅ **座標確認**: 相模大野駅の座標は両路線で一致 (35.5297, 139.4436)
4. 🔍 **進行中**: RouteFinder での駅登録状況をデバッグ中

**技術的分析:**
- 経路検索アルゴリズム（`src/utils/routeFinder.ts`）に詳細デバッグログを追加
- `debugStationRegistration()` メソッドで藤沢駅・相模大野駅の登録状況を確認中
- 相模大野駅での乗り換え検出ロジックにログ追加済み

**疑われる原因:**
1. **駅名マッピング問題**: `stationToRoutes` Map での駅名キーの不一致
2. **データ読み込み問題**: 小田急線データの読み込みエラー
3. **乗り換え検出問題**: 相模大野駅での路線間乗り換えが正しく認識されない
4. **探索優先度問題**: JRルートが優先されて小田急ルートが除外される

**デバッグ実装済み:**
- 初期化時の全路線・駅登録状況の出力
- 経路検索時の詳細ログ（出発駅・到着駅の路線情報）
- 相模大野駅での乗り換え検出ログ
- 最終推薦結果のサマリー表示

**次のアクションプラン:**
- ブラウザコンソールでの詳細ログ確認
- 小田急線データの読み込み状況検証
- 乗り換え探索アルゴリズムの修正（必要に応じて）

**ファイル:** `src/utils/routeFinder.ts`, `src/data/odakyu-*.ts`

## 開発者向け制御機能

### 路線表示切替セクションの表示制御

**概要:**
路線表示切替セクション（Route Display Toggle）は、デフォルトで非表示に設定されており、プログラムから動的に表示/非表示を制御できます。

**制御方法:**
```typescript
// RailwayMap.tsx 内で以下の状態変数を操作
const [showRouteToggleSection, setShowRouteToggleSection] = useState(false);

// 表示する場合
setShowRouteToggleSection(true);

// 非表示にする場合
setShowRouteToggleSection(false);
```

**含まれる機能:**
- 個別路線の表示/非表示切り替え
- 経路推薦数の設定（1, 2, 3, 5, 10件）
- 全路線表示/非表示の一括操作
- 時間フィルター設定（5分〜60分）
- デバッグ用テストボタン（山手線・中央線ポップアップ）

**実装場所:** `src/components/RailwayMap.tsx:64, 1222-1538`

**現在の状態:** デフォルトで非表示（showRouteToggleSection = false）

- 本番デプロイ（`--prod`）はユーザーの明示的な指示があった場合のみ実行する

### CHANGE.log 記録ルール

**セッション中に行った変更は必ず `CHANGE.log` に記録すること。**

- セッション終了時（またはまとまった作業が一区切りついたとき）に、そのセッションで行った変更を `CHANGE.log` に追記する
- フォーマットは既存のエントリに倣う（日付・セッション見出し → `###` 単位の変更概要）
- feat/fix/improve/docs のプレフィックスを使い、対象ファイルも明記する
- ユーザーから「CHANGE.logに記録して」と言われなくても、自発的に記録する

### コミットルール

**重要な開発ルール:**
1. **勝手にコミット・チェックアウトしないこと**
   - コミットする前には必ずユーザーに確認を求める
   - ユーザーが明示的にコミットを依頼した場合のみ実行する
   - 自動的なブランチ切り替えやチェックアウトは禁止

2. **論理的な単位でコミットすること**
   - **まとめてコミットしない**
   - 一つの論理的な変更単位ごとにコミットする
   - 関連のない変更は別々のコミットに分ける
   - 各コミットは単独で理解可能で、意味のある変更を含む

3. **コミットメッセージの品質**
   - 変更内容を簡潔に説明する
   - 何を変更したかではなく、なぜ変更したかを重視する
   - 日本語で記述し、機能追加は `feat:`、改善は `improve:`、修正は `fix:` で始める

**良いコミットの例:**
- 機能追加のコミット: `feat: 所要時間表示を数字のみの円形デザインに統一`
- バグ修正のコミット: `fix: 小田急線経路推薦が表示されない問題を修正`
- 改善のコミット: `improve: 駅枠線の二重・三重表示に白い間隔を追加`

**悪いコミットの例:**
- 複数の無関係な変更をまとめたコミット
- ユーザーの許可なく実行されたコミット
- 説明が不十分なコミットメッセージ
